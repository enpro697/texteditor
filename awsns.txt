### 正解選択肢D：Amazon SQSをデカップリングバッファとして導入し、AWS LambdaがメッセージをポーリングしてDynamoDBとAmazon SNSへ非同期処理を行う

<竜胆> Eliminate the impossible, and whatever remains must be the truth. Behold, Option D: Implement Amazon Simple Queue Service <SQS> as a decoupling buffer between the EC2 web tier and the database tier. Configure an AWS Lambda function to poll the SQS queue, process the messages, write to DynamoDB, and publish to Amazon SNS.
<竜胆> 不可能な選択肢を排除していけば、最後に残ったものが真実となる。見よ、選択肢Dだ：「EC2ウェブ層とデータベース層の間に、デカップリングバッファとしてAmazon Simple Queue Service（SQS）を導入する。AWS Lambda関数を設定してSQSキューをポーリングさせ、メッセージを処理し、DynamoDBへの書き込みとAmazon SNSへのパブリッシュを非同期で行う」。

<惟緒> Okay, Hinorin, let's unpack this slowly. What is SQS? And why is it better than Kinesis or Aurora or whatever?
<惟緒> オッケー、ヒノりん、ゆっくり紐解いていこう。SQSってそもそも何？なんでそれが、KinesisとかAuroraとかより優れてるの？

<竜胆> Amazon SQS—Simple Queue Service—is a fully managed message queuing service designed precisely for decoupling and scaling distributed systems. Imagine an infinitely deep, perfectly organized inbox. When an EC2 instance processes a user request, it no longer attempts to write directly to DynamoDB. Instead, it simply packages the data as a message and drops it into the SQS queue. The EC2 instance considers its job done immediately and returns a success response to the user.
<竜胆> Amazon SQS——Simple Queue Service——は、分散システムの疎結合化とスケーリングのために正確に設計された、フルマネージド型のメッセージキューイングサービスだ。無限の深さを持つ、完璧に整理整頓された受信トレイを想像してみよ。EC2インスタンスがユーザーからのリクエストを処理した際、もはやDynamoDBへ直接書き込もうとはしない。代わりに、データをメッセージとしてパッケージ化し、SQSキューへ投函するだけだ。EC2インスタンスは即座に自らの仕事を完了したとみなし、ユーザーへ成功レスポンスを返すのだ。

<惟緒> That makes so much sense! The web servers don't care if the database is busy or having a panic attack. They just drop the message into SQS and walk away like cool guys in an action movie walking away from an explosion!
<惟緒> めちゃくちゃ腑に落ちた！ウェブサーバーたちは、データベースが忙しかったりパニック発作を起こしてたりしても一切気にしないわけだ。ただSQSにメッセージをポイって入れて、アクション映画のクールな主人公みたいに爆破から歩き去るだけ！

<竜胆> A dramatic visual, but technically sound. This completely resolves the tight coupling. But now, how does the data get from the SQS queue into DynamoDB and SNS? This is where AWS Lambda enters the architecture.
<竜胆> 劇的な視覚イメージだが、技術的には正当だ。これにより密結合は完全に解消される。では、そのデータはどうやってSQSキューからDynamoDBとSNSへと運ばれるのか？ここでアーキテクチャに登場するのが、AWS Lambdaだ。

<惟緒> Oh, Lambda! I've heard of that! It's the ultimate serverless superstar, right? How does it interact with SQS?
<惟緒> あ、Lambda！それ聞いたことある！究極のサーバーレスのスーパースターでしょ？それがSQSとどう関わるの？

<竜胆> AWS Lambda is an event-driven, serverless compute service. You do not provision any servers. Instead, we configure the Lambda function to monitor—or "poll"—the SQS queue. When messages appear in the queue, AWS automatically spins up Lambda execution environments to retrieve the messages, execute your business logic, write the records to DynamoDB, and send the required payloads to Amazon SNS.
<竜胆> AWS Lambdaは、イベント駆動型のサーバーレスコンピューティングサービスだ。サーバーを一切プロビジョニングする必要はない。その代わり、SQSキューを監視——あるいは「ポーリング」——するようにLambda関数を設定するのだ。キューにメッセージが出現すると、AWSは自動的にLambdaの実行環境を立ち上げ、メッセージを取得し、業務ロジックを実行し、DynamoDBへレコードを書き込み、必要なペイロードをAmazon SNSへと送信する。

<惟緒> English, Hinorin! What does setting a "concurrency limit" mean for our ghost workers?
<惟緒> ヒノりん、日本語で頼む！「同時実行数の制限」を設定するって、私たちの幽霊労働者たちにとってどういう意味になるの？

<竜胆> それは、お前が最高司令官として、Lambdaサービスにこう命令を下すことを意味する：「SQSのロビーで待機するメッセージが何百万件あろうとも、単一の瞬間において50体以上の幽霊労働者を同時に生み出してはならない」とな。処理速度がDynamoDBの設定済みキャパシティに合致するように人為的に制限されるため、データベースが圧倒されることは絶対にない。常に安定した、管理可能なデータ流を、自らの最適なペースで処理し続けるのだ。

<竜胆> When a message fails to be processed after a specified maximum number of attempts—say, three times—SQS automatically ejects it from the main queue and moves it into the Dead-Letter Queue. Engineers can then safely inspect the quarantined messages in the DLQ at their leisure without disrupting production.
<竜胆> 指定された最大試行回数——例えば3回——を経ても処理に失敗した場合、SQSはそのメッセージをメインのキューから自動的に排出し、デッドレターキューへと移送するのだ。これにより、問題のあるデータを隔離し、通常のトラフィック処理が一切阻害されることなく継続するようにする。エンジニアはその後、本番環境に影響を与えることなく、余裕を持ってDLQ内に隔離されたメッセージを安全に調査できるということだ。

<怜空> Ah, TCP. The bureaucratic antithesis to UDP’s reckless freedom. TCP is a stateful, connection-oriented protocol. <竜胆> お前の深夜のインターネット習慣はさておき、HTTPは確かにウェブの共通言語であり、マイクロサービスにおいてはRESTful APIの骨格を形成している。**HTTP/1.1**は、伝統的で広く普及している標準規格だ。堅牢でどこにでもあるが、構造的には直列的だ。スーパーマーケットの1レーンしかないレジを想像してみるがよい。一人の客が怪しい缶詰の価格確認を始めたら、後ろに並んでいる列全体が苦痛に満ちた停止状態に陥る。これは「ヘッド・オブ・ライン・ブロッキング」として知られている現象だ。App Meshは、既存のレガシーなマイクロサービスが複雑なコードの書き換えを伴わずにメッシュに参加できるよう、HTTP/1.1をサポートしているのだ。

<怜空> And then, Iochi, we arrive at **HTTP/2**—the evolution forged in the fires of impatience.
